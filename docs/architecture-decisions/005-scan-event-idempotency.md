# ADR 005: Idempotency Strategy for Custody Scan Events

## Context

A courier's scan can be submitted more than once for reasons that have nothing to do with a genuine second event: a double-tap on the device, a network timeout where the courier doesn't know if the first submission succeeded and retries, or an offline-first client replaying a queued scan after connectivity returns. If each of these creates a second `CourierAccepted` event, the custody log gets a duplicate entry, the SLA timer resets incorrectly, and Ops potentially receives a duplicate notification, undermining the exact reliability story this project exists to build.

EventBridge itself doesn't solve this. It provides at-least-once delivery with no built-in deduplication, so any duplicate that reaches the bus is treated as a genuine new event by every downstream consumer.

## Decision

Every scan event carries a client-generated idempotency key, a UUID generated on the courier's device at the moment of the scan, not assigned by the server. The persist step (per ADR 003's validate, persist, publish, notify, audit pipeline) performs a conditional write against the custody log using `ConditionExpression: attribute_not_exists(idempotency_key)`: if an item with that idempotency key already exists, DynamoDB rejects the write with a `ConditionalCheckFailedException`, no new custody log entry is created, and nothing gets published to EventBridge. Deduplication happens before publish, not after, so duplicate events never reach downstream consumers at all.

## Reasoning

The idempotency key has to be generated by the client, not the server, because the failure case that matters most is the client not knowing whether its first request succeeded. If the server generated a new identifier on every request, a legitimate retry would look identical to a brand-new event, and the whole point of the key is lost. A client-generated key means the courier's device can safely resubmit the exact same scan as many times as needed, and the system will only ever record it once.

Performing the dedup check as a conditional write on the persist step, rather than as a separate service or a distributed lock, keeps this simple and consistent with the existing custody log as the system of record. It also means the guarantee holds regardless of what happens downstream: since duplicates are caught before publish, no EventBridge consumer, not the SLA timer, not the notification service, ever needs its own separate deduplication logic.

## Alternatives Considered

**Deduplicate by content hash instead of a client-generated key.** Rejected: a legitimate correction, the same specimen genuinely re-scanned on purpose, could hash identically to an accidental duplicate, and content hashing can't distinguish the two. A client-generated key tied to a specific scan action doesn't have this ambiguity.

**Rely on SQS FIFO's built-in `MessageDeduplicationId`.** Rejected: this project uses EventBridge as its backbone (ADR 003), not SQS, and EventBridge has no equivalent native mechanism. Introducing SQS purely for its dedup window would add a second messaging system for one feature.

**Server-generated idempotency key.** Rejected: doesn't solve the retry case. A server-assigned ID changes on every request, so the server can't recognize a retry as the same logical event.

## Consequences

**Positive**
- A courier can safely retry any scan without risk of creating a duplicate custody record
- Offline-first mobile behavior becomes possible: queued scans can replay safely once connectivity returns, since retries are inherently safe
- No downstream consumer needs its own deduplication logic

**Negative**
- Every event now requires an additional client-generated field, adding a small amount of complexity to the mobile/scanning client
- Idempotency keys need a retention policy; deduplicating forever against an unbounded key store isn't sustainable, so a reasonable time window needs to be defined once the client is built
- A genuine intentional re-scan (a correction, not a retry) needs to be modeled as an explicit new event type rather than relying on the idempotency key alone, this project does not yet define that distinction

## Revisit If

A legitimate need arises to distinguish an intentional correction from an accidental duplicate, at that point a distinct `CorrectionRecorded` event type should be introduced rather than stretching the idempotency key to serve both purposes.
